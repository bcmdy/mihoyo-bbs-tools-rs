# 安全说明

本文档规定 MihoyoBBSTools RS 在配置、日志、网络访问、测试和发布过程中的最低安全要求。任何业务功能只有满足这些要求后才能视为完成。

## 凭据管理

- 真实 Cookie、SToken、LToken、Cookie Token、推送 Token、Webhook 密钥、SMTP 凭据和代理密码只能通过 GitHub Secrets、环境变量或受控配置文件提供。
- 仓库中的示例配置只允许使用 `${ENV_NAME}` 占位符。
- 不允许将 Cookie 或 Token 作为命令行参数，避免凭据出现在进程列表、Shell 历史和 Actions 日志中。
- 敏感字段在 Rust 中应使用敏感字符串类型包装，不能派生或实现会输出明文的普通 `Debug`。
- 凭据刷新流程不得把新凭据写入日志。未来如支持写回配置，必须采用显式开关、安全文件权限和原子替换。

## 日志脱敏

以下内容必须在进入日志、错误报告、任务汇总或推送消息前脱敏：

- `Cookie` 与 `Authorization` 请求头。
- SToken、LToken、Cookie Token 和其他登录票据。
- Telegram Bot Token 及其他推送提供商令牌。
- Webhook URL 查询参数或路径中的密钥。
- 代理用户名和密码。
- SMTP 用户名和授权码或密码。
- 完整账号 ID，默认只显示尾部少量字符。

禁止记录完整的 HTTP 请求对象、配置结构体、环境变量集合或未经限制的响应体。错误信息只能包含定位问题所需的安全上下文。

## 网络安全

- 默认验证 TLS 证书，不提供全局关闭证书验证的选项。
- SMTP 默认使用隐式 TLS；`tls: none` 只允许用于可信隔离网络，因为认证信息会以明文传输。
- Windows 本地通知通过固定脚本调用系统 WinRT，标题和正文只能通过子进程环境变量传递，不得拼接进 PowerShell 命令文本。
- 验证码服务地址、代理地址和 Webhook 地址必须校验协议。
- Cookie 和认证请求头只能发送给明确允许的目标主机。
- HTTP 重定向到不同主机时，不得继续携带敏感请求头。
- 所有请求必须设置超时，响应体必须设置大小上限。
- 只有连接失败、超时和经过确认的部分 5xx 请求可以重试。
- 签到等可能产生副作用的 POST 请求不得盲目重试，应先查询状态或使用业务幂等判断。

## GitHub Actions

- 普通 CI 使用最小权限 `contents: read`。
- 真实接口测试不能在 Pull Request 中自动运行，也不能向来自 Fork 的工作流提供 Secret。
- 真实接口测试应使用独立的 `integration` Environment，并配置人工批准。
- Actions 不得上传真实配置、运行日志、账号信息或其他含 Secret 的 Artifact。
- Release 任务仅在版本标签触发时，为创建 GitHub Release 的作业单独授予 `contents: write`。

## 测试数据

- 单元测试和 HTTP 合约测试不得访问真实米哈游或 HoYoLAB 接口。
- Fixture 必须经过人工脱敏，不得包含真实 Cookie、Token、账号 ID、代理地址或历史日志。
- 签名测试使用固定的虚构测试向量。
- 真实接口验证先执行只读状态查询，再逐步开放实际签到请求。

## Docker 与发布

- Secret 不得通过 `ARG`、`ENV`、`COPY` 或镜像层写入 Docker 镜像。
- 最终镜像使用非 root 用户运行，只包含可执行文件、CA 证书和时区数据。
- 发布产物必须提供 SHA256 校验文件。
- 发布前应确认 Git 历史和构建产物中不存在真实凭据。
- DaCapo JSON 只允许在内存中转换，不得生成或上传含 Cookie、Token 的临时 YAML、INI 或调试快照。

## 安全问题报告

发现凭据泄露时，应立即撤销相关凭据，停止对应工作流并清理可下载的构建产物。由于 Git 历史中的 Secret 不能通过普通删除可靠撤回，任何已经提交的真实凭据都必须视为已泄露并立即轮换。
