# MihoyoBBSTools RS 使用说明

本说明面向直接使用程序的用户，适用于 Windows 与 Linux 发布包。第一次使用先阅读下面的“快速开始”；通知、自动运行和高级集成可在需要时再查阅对应章节。

## 快速开始

1. 从 [GitHub Releases](https://github.com/bcmdy/mihoyo-bbs-tools-rs/releases/latest) 下载对应平台的压缩包，并完整解压到固定目录。
2. 在解压目录打开 PowerShell、Windows 终端或 Linux 终端。
3. 创建配置并运行：

Windows：

```powershell
.\MihoyoBBSToolsRS.exe config init
.\MihoyoBBSToolsRS.exe run
```

Linux：

```bash
chmod +x MihoyoBBSToolsRS
./MihoyoBBSToolsRS config init
./MihoyoBBSToolsRS run
```

`config init` 会用 `*` 掩码显示 Cookie 输入，并引导选择账号、游戏、米游社板块和可选通知。首次创建的账号默认只执行原神国内游戏签到，以及大别野和原神社区签到；阅读、点赞、分享、HoYoLAB、云游戏和 Web 活动默认关闭。

配置保存在 `config/config.yaml`。以后通常只需执行 `run`；修改设置使用 `config setup`，添加账号使用 `config add-account`，检查配置使用 `validate-config`。请勿把 Cookie、SToken、通知 Token 或完整配置发到公开位置。

阅读导航：

| 需求 | 章节 |
|---|---|
| 下载、添加 Cookie、首次运行 | 本页开头的“快速开始”及第 2–5 节 |
| 多账号、任务开关、命令 | 第 6–9 节 |
| 验证码、代理、通知 | 第 10–12 节 |
| YAML 全部字段 | 第 13 节 |
| 日志与自动运行 | 第 14–15 节 |
| Cookie 字段与凭据故障 | 第 4 节及第 18 节 |
| 旧配置迁移、状态码、故障排查与安全 | 第 16–20 节 |

普通用户优先使用 `config init`、`config setup` 和 `run`。`run-directory`、青龙、DaCapo、Docker、Kubernetes 与 Nix 都是可选的高级用法，不影响桌面环境的正常使用。

## 1. 软件用途

MihoyoBBSTools RS 可以按配置执行以下任务：

- 米哈游国内游戏签到。
- HoYoLAB 国际服游戏签到。
- 米游社社区签到。
- 可选的阅读、点赞、取消点赞和分享任务。
- 国内云原神、云绝区零和国际服云原神签到。
- 多账号依次运行并生成统一结果报告。
- 验证码平台接入。
- Telegram、Webhook、PushPlus 等多种通知。
- 账号代理及 Telegram 独立代理。
- 按日保存运行日志。

这是社区维护的非官方工具，与米哈游、HoYoverse 及其关联公司没有隶属、授权或合作关系。

## 2. 发布包内容

解压后目录如下：

```text
MihoyoBBSToolsRS-<版本>-<平台>-<架构>/
├── MihoyoBBSToolsRS[.exe]
├── 使用说明.md
├── config/
│   └── config.example.yaml
└── dacapo/
    └── template.yml
```

Windows 使用 `MihoyoBBSToolsRS.exe`，Linux 使用 `MihoyoBBSToolsRS`。Linux 首次运行前可能需要授予执行权限：

```bash
chmod +x MihoyoBBSToolsRS
```

正式标签构建的压缩包名称类似：

```text
MihoyoBBSToolsRS-<版本>-windows-x86_64.zip
MihoyoBBSToolsRS-<版本>-linux-arm64.tar.gz
```

压缩包和 Actions Artifact 带版本号，但包内可执行文件始终固定为 `MihoyoBBSToolsRS.exe` 或 `MihoyoBBSToolsRS`，升级时直接覆盖旧程序即可。可通过 `-V`、`--version` 或 `version` 命令确认实际构建版本。

命令中的相对路径以当前工作目录为基准。建议始终在程序所在目录运行。

需要测试 `main` 最新代码时，可以在仓库的 Actions 页面手动运行“Rust CI”，在 `version` 输入测试版本号，完成后从该次任务页面底部的 Artifacts 下载对应平台压缩包。普通 CI 产物不是正式 Release；创建正式版本应先推送对应的 `v<版本>` 标签，再运行“发布”工作流或等待标签自动触发。

## 3. 首次配置说明

执行 `config init` 后按向导输入完整 Cookie。交互终端会按 Unicode 字符显示等量 `*`，既能确认键入或粘贴已生效，也不会回显明文。程序会：

1. 校验 Cookie 格式并提取 UID 与 SToken。
2. 查询米游社昵称。
3. 使用 `mys用户:<米游社昵称>` 作为配置中的账号名称。
4. 选择游戏、任务、米游社板块和可选通知渠道。
5. 显示不含 Secret 的最终摘要，确认后一次创建 `config/config.yaml`。
6. 可选发送测试通知、执行第一次运行和安装 Windows 每日任务。

已有配置需要添加账号时使用 `config add-account --name "备注"`。`config init` 检测到目标配置已经存在时会拒绝覆盖，并提示改用 `config setup`。

新账号默认只启用：

- 原神国内游戏签到。
- 米游社社区签到。

阅读、点赞、取消点赞、分享、HoYoLAB、云游戏和 Web 活动默认关闭。

## 4. Cookie 要求与安全

Cookie 必须完整，并至少能识别 UID、SToken；V2 SToken 还需要同一登录会话的 MID。程序会自动提取，不需要手工拆分填写。国内账号常见字段如下：

| 用途 | 常见字段 | 要求 |
|---|---|---|
| 账号 UID | `account_id`、`account_id_v2`、`ltuid`、`ltuid_v2`、`stuid` | 至少存在一个有效数字 UID |
| 社区任务与刷新 | `stoken` | 添加国内账号时必须存在 |
| V2 身份 | `account_mid_v2`、`ltmid_v2`、`mid` | `stoken` 以 `v2_` 开头时必须存在 |
| 游戏签到与社区请求 | `cookie_token` | 失效时程序可尝试通过 SToken 刷新 |
| 旧式登录流程 | `login_ticket` | 仅部分旧会话存在，不是必填字段 |

“游戏签到当前能用”不代表 Cookie 完整；只含 UID 和 `cookie_token` 的内容可能无法执行社区任务、昵称查询或凭据刷新。若提示“已识别 UID，但缺少 SToken”或“V2 SToken 缺少 MID”，应重新获取完整 Cookie，不要从其他账号拼接字段。HoYoLAB 使用独立 Cookie，应在账号的 HoYoLAB 设置中录入。

程序不会自动读取浏览器 Cookie。只从自己信任的登录会话获取，不要安装来源不明的 Cookie 导出扩展、脚本或修改版程序。

安全建议：

- 不要把 Cookie、SToken、通知 Token、Webhook 或代理密码发到聊天、截图或公开仓库。
- 不要把 Cookie 直接写进命令参数，以免进入终端历史或进程列表。
- 推荐使用 `config add-account` 或 `config setup` 输入 Cookie。
- 配置和日志应只允许可信用户读取。
- Cookie 失效后使用设置菜单的“更新 Cookie”，程序会同时更新 SToken 与昵称。

Cookie、Token、密码、代理凭据和含访问凭据的端点在交互输入时都显示 `*`。退格会同步擦除星号；按 Enter 后会立即提示已接收，再执行格式校验或联网操作。已保存的值只显示 `<已配置>`，不泄露原值或长度；非交互标准输入保持静默。若检测到内容前后两半完全相同，会给出不含原文的疑似重复粘贴提示并要求重新输入。

推荐使用以下命令排查凭据，不要把真实 Cookie 放进命令行：

```powershell
.\MihoyoBBSToolsRS.exe validate-config
.\MihoyoBBSToolsRS.exe doctor
.\MihoyoBBSToolsRS.exe doctor --online
.\MihoyoBBSToolsRS.exe run --verbose
```

## 5. 配置文件

默认配置路径为：

```text
config/config.yaml
```

可以通过 `--config` 指定其他文件：

```powershell
.\MihoyoBBSToolsRS.exe run --config D:\secure\mihoyo.yaml
```

配置有四种方式：

1. `config init`：首次创建完整配置，不覆盖已有文件。
2. `config add-account`：向已有配置快速添加账号。
3. `config setup`：推荐的完整数字菜单，可修改全部常用配置。
4. `config edit`：使用系统编辑器直接修改 YAML，适合熟悉 YAML 的高级用户。

也可以复制模板：

```powershell
Copy-Item config\config.example.yaml config\config.yaml
```

复制后必须替换示例 Cookie 或环境变量占位符。

每次修改后建议执行：

```powershell
.\MihoyoBBSToolsRS.exe validate-config
```

## 6. 完整设置菜单

进入菜单：

```powershell
.\MihoyoBBSToolsRS.exe config setup
```

一级菜单包括：

- 全局运行：请求超时、重试、随机延迟、日志级别、文件日志、定时运行开关与间隔。
- 验证码：验证码平台端点。
- 账号：添加、删除、启用状态、备注、Cookie、设备、代理、国内签到、HoYoLAB、云游戏、任务、米游社社区板块、国内游戏。
- 通知：通用选项及通知渠道的添加、编辑、删除。
- 校验：立即检查当前配置。
- 高级 YAML 编辑：可选的完整文件编辑入口。

菜单输入规则：

- 留空通常表示保留当前值或取消当前操作。
- `-` 表示清空可选字段。
- `0` 表示返回上一级。
- 多选可以输入连续数字 `123`，也可以输入 `1,2,3`。
- 带 `[x]` / `[ ]` 的任务、游戏和板块菜单中，输入编号表示切换；`a` 全选、`n` 全不选、`s` 保存当前子菜单。
- Token、Cookie 与代理认证信息不会作为当前值回显；新输入按字符显示 `*`，按 Enter 后立即确认已接收。

所有菜单操作先修改受控临时副本。选择“保存并退出”后，程序显示脱敏变更摘要；只有确认后才完整校验、备份旧配置并原子替换。选择取消、输入结束、读取失败或校验失败时，真实配置保持不变。删除账号和通知渠道还需要再次输入 `yes` 或对应编号。

## 7. 多账号

每个账号都有独立的：

- `name` 与 `remark`。
- 启用状态。
- Cookie 与 SToken。
- 国内签到 User-Agent 与各游戏角色 UID 黑名单。
- HoYoLAB 独立 Cookie、语言、User-Agent 与游戏列表。
- 设备名称、型号、ID、FP。
- 网络代理。
- 任务开关。
- 游戏列表。

程序按配置顺序运行账号。账号名称一般为 `mys用户:<昵称>`；昵称重复时会追加 UID 尾号。备注只用于帮助用户区分账号。

添加账号：

```powershell
.\MihoyoBBSToolsRS.exe config add-account --name "小号"
```

删除账号时使用配置中的账号名称：

```powershell
.\MihoyoBBSToolsRS.exe config remove-account "mys用户:旅行者"
```

程序会再次显示账号名称和备注并要求确认；非交互脚本必须显式增加 `--yes`。

## 8. 任务与游戏

主要任务字段：

| 配置项 | 用途 |
|---|---|
| `china_game_checkin` | 国内游戏签到 |
| `hoyolab_checkin` | HoYoLAB 国际服签到 |
| `bbs.enabled` | 米游社任务总开关 |
| `bbs.sign` | 米游社社区签到 |
| `bbs.forums` | 社区板块列表；顺序首项也用于获取帖子，可在设置菜单多选 |
| `bbs.read` | 阅读帖子 |
| `bbs.like` | 点赞帖子 |
| `bbs.cancel_like` | 点赞后取消点赞 |
| `bbs.share` | 分享帖子 |
| `china_cloud_game` | 国内云游戏总开关；还需启用 `cloud_games.china` 下的具体游戏 |
| `overseas_cloud_game` | 国际服云游戏总开关；还需启用 `cloud_games.overseas.genshin` |
| `web_activity.enabled` | Web 活动总开关 |
| `web_activity.activities` | Web 活动标识；当前仅识别已结束的 `genshin_mizone` |

游戏值：

| 值 | 游戏 |
|---|---|
| `genshin` | 原神 |
| `honkai2` | 崩坏学园2 |
| `honkai3rd` | 崩坏3 |
| `tears_of_themis` | 未定事件簿 |
| `star_rail` | 崩坏：星穹铁道 |
| `zenless_zone_zero` | 绝区零 |

没有绑定角色的游戏在默认文本报告中显示“已跳过”，不会视为运行失败；JSON 中仍使用稳定枚举 `skipped`。

云游戏在每个账号的 `cloud_games` 下配置。国内的 `genshin`、`zenless_zone_zero` 和国际服的 `genshin` 都有 `enabled` 与 `token`；国际服还可将 `language` 设为 `zh-cn`、`en-us`、`ja-jp` 或 `ko-kr`。Token 属于敏感凭据，`null` 表示未配置；交互设置不会回显已保存的值。Token 已保存但暂时不执行时，只需关闭对应的 `enabled`，无需删除 Token。

国内云游戏首次未显示新增时长且总时长低于 600 分钟时会等待 3–6 秒复查，以总时长增量确认到账。Token 失效和防沉迷限制可能使用相同错误码，程序不会自动删除 Token；报告认证失败后请自行确认并更新配置。

游戏签到每次提交后都会复查对应角色。只有复查仍显示今日未签到、奖励仍可领取时，程序才重新提交这个角色；已确认签到的角色不会重复执行。默认最多尝试 `3` 次，可通过 `runtime.task_max_attempts` 调整为 `1`–`10`。如果复查接口本身失败，程序会停止该角色的后续提交，避免状态未知时重复签到。旧字段 `runtime.game_checkin_max_attempts` 仍可读取，并按相同数值迁移到新字段。

|---|---|

只运行指定任务：

```powershell
.\MihoyoBBSToolsRS.exe run --task china-checkin
.\MihoyoBBSToolsRS.exe run --task china-checkin,bbs
.\MihoyoBBSToolsRS.exe run --task china-cloud-game,overseas-cloud-game
```

任务值为 `china-checkin`、`hoyolab-checkin`、`bbs`、`china-cloud-game`、`overseas-cloud-game`、`web-activity`。命令行筛选只会缩小本次范围，不会重新启用 YAML 中已经关闭的任务。

只读或自动化调用：

```powershell
Get-Content -Raw D:\secure\mihoyo.yaml | .\MihoyoBBSToolsRS.exe run --config - --read-only --no-notify --output json
.\MihoyoBBSToolsRS.exe run --config D:\secure\mihoyo.yaml --read-only --output json
```

- `--config -`：从标准输入读取完整 YAML，自动采用只读模式。
- `--read-only`：允许凭据在本次运行的内存中刷新，但禁止改写配置文件。
- `--no-notify`：不调用任何通知渠道，JSON 中的 `notifications` 为空数组。
- `--output text|json`：默认 `text`；`json` 时标准输出只包含一个 JSON 对象，日志进入标准错误或文件日志。
- `--verbose`：仅影响文本模式，在普通摘要之后展开全部逐项记录；JSON 不变。

默认文本报告先显示“成功、今日已完成、跳过、失败”的计数，再按账号展示关键成功项、跳过项和需要处理的问题。游戏签到包含脱敏角色、累计天数与今日奖励；社区任务包含米游币已领取、还可领取和当前总数；阅读、点赞、取消点赞和分享只按任务类型聚合。INFO 文件日志与通知使用同一摘要，`run --verbose` 和 JSON 保留逐项记录。

诊断示例：

```powershell
.\MihoyoBBSToolsRS.exe doctor
.\MihoyoBBSToolsRS.exe doctor --online
.\MihoyoBBSToolsRS.exe doctor --output json
```

离线模式不访问网络。在线模式会明确发起只读请求，但不会签到、阅读、点赞、分享、领取云游戏时长或发送通知。诊断 JSON 使用独立的 `schema_version`、`online` 和 `items` 结构，不与运行报告混用。

JSON 顶层固定包含 `schema_version`、`exit_code`、`records` 和 `notifications`。`records` 的每项包含 `account`、`task`、`subject`、`outcome`、`message`；通知项包含 `provider`、`status`、`message`。字段名和枚举值使用 snake_case，当前 `schema_version` 为 `1`。

多配置目录运行：

```powershell
.\MihoyoBBSToolsRS.exe run-directory config
.\MihoyoBBSToolsRS.exe run-directory D:\configs --prefix mhy_
.\MihoyoBBSToolsRS.exe run-directory D:\configs --task china-checkin,bbs
.\MihoyoBBSToolsRS.exe run-directory D:\configs --delay-min-seconds 0 --delay-max-seconds 0
```

目录模式只读取当前目录，不递归子目录；支持 `.yaml` 和 `.yml`，按文件名确定性排序，并自动排除 `*.example.yaml`、`*.example.yml`。`--prefix` 按文件名区分大小写筛选。单个文件损坏、缺少环境变量或任务失败时会记录到批次汇总并继续下一个文件；每个成功加载的配置使用自己的通知设置。文件之间默认随机等待 3–10 秒，范围必须满足 `0 <= 最小值 <= 最大值 <= 3600`。

批次退出码按语义聚合：内部错误、认证失败、验证码、网络、配置和普通任务失败依次优先；任一文件非成功时批次返回非零。多个文件中可使用相同账号名称，汇总仍会保留配置文件名用于区分。为避免同时写回凭据，目录模式保持串行执行。

青龙兼容入口：

```bash
./MihoyoBBSToolsRS qinglong
./MihoyoBBSToolsRS ql --task china-checkin,bbs
```

支持的环境变量：

| 环境变量 | 作用 |
|---|---|
| `AutoMihoyoBBS_config_path` | 配置目录，默认 `config`。 |
| `AutoMihoyoBBS_config_prefix` | 配置文件名前缀；单配置模式读取 `<前缀>config.yaml`。 |
| `AutoMihoyoBBS_config_multi` | `0` 或未设置为单配置，`1` 为多配置；其他值直接报错。 |

DaCapo 运行（可选高级功能）：

DaCapo 是第三方 JSON 配置适配入口，不是签到任务或必需组件。没有在使用 DaCapo 的用户可以忽略本段、`dacapo` 命令和发布包中的 `dacapo` 目录。

```powershell

.\MihoyoBBSToolsRS.exe dacapo D:\dacapo\settings.json --task china-checkin,bbs
```

发布包内提供 `dacapo/template.yml`。DaCapo 应将生成的 JSON 文件路径追加到模板命令 `./MihoyoBBSToolsRS dacapo` 后；也可手动执行上面的命令验证。适配器同时接受直接值和 `{ "value": ... }`：

```json
{
  "Project": {
    "General": {
      "账号配置": {
        "米游社Cookie": { "value": "${MIHOYO_COOKIE}" },
        "stuid": "123456789",
        "stoken": { "value": "${MIHOYO_STOKEN}" },
```

## 10. 验证码平台

在 `config setup → 验证码` 中填写平台端点，或配置：

```yaml
captcha:
  endpoint: "https://example.com/captcha"
```

不使用验证码平台时设置为 `null`。端点必须是完整的 HTTP/HTTPS 求解地址，程序不会自动追加路径。服务需要兼容 pass_nine 的 GET 协议：接收 `gt`、`challenge`、`use_v3_model=true`，并在顶层或 `data` 中返回 `validate` 和可选的 `challenge`。

国内游戏签到遇到验证码时会求解并携带结果重新提交；社区签到、点赞和取消点赞会按“创建验证 → 平台求解 → 服务端校验 → 重试原操作”的顺序处理。端点可能包含访问凭据，应按敏感信息处理。

社区任务调用验证码平台遇到请求超时、连接失败、非成功 HTTP 状态、无效 JSON、缺少非空 `validate` 或 `result: fail` 等异常时，会等待 3 秒并重新调用米游社 `createVerification`，使用全新的 `gt/challenge` 再次求解，最多执行 `runtime.task_max_attempts` 轮。验证码平台请求本身不再叠加通用 HTTP 自动重试，避免请求次数相乘；最后一轮失败后不再等待。米游社创建或校验验证码明确失败、以及校验成功后原动作仍要求验证码时会立即停止高风险动作。

遇到验证码时，结果可能显示 `CaptchaRequired`。它表示未配置平台、求解失败、服务返回失败结果，或重试后仍要求验证码；程序不会无限重试，也不会把动作接口的成功响应伪装成最终完成。

### 12.1 Telegram 详细设置

#### 创建机器人并取得 Token

1. 在 Telegram 中搜索官方机器人 `@BotFather`。
2. 发送 `/newbot`。
3. 按提示填写机器人显示名称。
4. 填写一个以 `bot` 结尾的机器人用户名。
5. 保存 BotFather 返回的 Bot Token，格式通常类似 `123456789:AA...`。

Bot Token 等同机器人密码。泄露后应立即在 BotFather 中使用 `/revoke` 重置，不要把包含 Token 的 URL、截图或配置发送给他人。

#### 获取私聊 Chat ID

1. 打开刚创建的机器人。
2. 点击 Start，或发送 `/start`。
3. 在浏览器访问：

   ```text
   https://api.telegram.org/bot<你的Token>/getUpdates
   ```

4. 在返回内容的 `result[].message.chat.id` 中找到数字 ID。

私聊用户必须先主动联系机器人，机器人不能主动发起第一次会话。

#### 获取群组 Chat ID

1. 将机器人加入目标群组，并确保机器人有发送消息权限。
2. 在群中发送 `/start@机器人用户名`。
3. 再次访问 `getUpdates`。
4. 读取 `message.chat.id`。群组 ID 通常是负数，超级群常以 `-100` 开头，必须连同负号完整填写。

如果 `result` 为空，可重新发送一条针对机器人的命令再刷新。机器人已被其他程序持续轮询或配置 Webhook 时，`getUpdates` 可能取不到消息；建议为本工具创建独立机器人。

#### Telegram 配置示例

```yaml
notifications:
  enabled: true
  error_only: false
  block_keywords: []
  providers:
    - type: telegram
      bot_token: "${TELEGRAM_BOT_TOKEN}"
      chat_id: "${TELEGRAM_CHAT_ID}"
      api_url: "https://api.telegram.org"
      proxy: null
```

字段说明：

| 字段 | 是否必填 | 中文说明 |
|---|---:|---|
| `type` | 是 | 固定填写 `telegram`。 |
| `bot_token` | 是 | BotFather 提供的机器人 Token。 |
| `chat_id` | 是 | 私聊、群组或频道 ID。建议始终用引号包裹，负数群组 ID 不能漏掉负号。 |
| `api_url` | 否 | Bot API 根地址，默认 `https://api.telegram.org`。不要填写 `/bot.../sendMessage` 完整地址。 |
| `proxy` | 否 | Telegram 独立代理；不用时为 `null`。 |

国内网络使用代理示例：

```yaml
proxy: "http://127.0.0.1:7890"
```

或：

```yaml
proxy: "socks5h://127.0.0.1:1080"
```

`socks5h` 会让域名解析也经过代理。带用户名和密码的代理可写成 `http://user:password@host:port`；用户名或密码中的 `@`、`:`、`/`、`#` 等字符必须进行 URL 编码。

只有使用 Telegram Bot API 反向代理或兼容网关时才修改 `api_url`。自定义地址必须兼容程序自动请求的 `/bot<TOKEN>/sendMessage` 路径。

设置菜单路径：

```text
config setup → 通知 → 添加渠道 → telegram
```

依次填写 `bot_token`、`chat_id`；`api_url` 直接回车采用默认值；不使用代理时 `proxy` 留空。添加渠道后返回“通用选项”启用通知。

常见问题：

- HTTP 400：检查 Chat ID、群组负号、群权限及是否已向机器人发送 `/start`。
- HTTP 401/404：Token 错误、Token 已重置，或 `api_url` 填写错误。
- HTTP 403：用户屏蔽机器人，或机器人被移出群、禁言、没有发送权限。
- HTTP 429：发送过于频繁，稍后再试。
- 网络失败：代理没有启动、端口或协议错误，或当前网络无法访问 Telegram。
- 配置正确但没有推送：检查 `notifications.enabled`；`error_only: true` 时成功运行不会发送通知。

程序当前发送纯文本，不支持 Telegram Markdown/HTML、消息主题 `thread_id` 或论坛话题选择。

### 12.2 微信通知渠道选择

可送达到微信或企业微信的渠道包括：

| 类型 | 渠道 | 适用场景 |
|---|---|---|
| `wecom` | 企业微信自建应用 | 给企业成员发送应用消息，适合组织内部使用。 |
| `wecomrobot` | 企业微信群机器人 | 向固定企业微信群发送消息，设置最简单。 |
| `wxpusher` | WxPusher | 通过微信公众号向个人 UID 或主题订阅者推送。 |
| `pushplus` | PushPlus | 通过微信公众号向个人或群组推送。 |
| `ftqq` | Server酱 Turbo | 使用 SendKey 推送到已绑定的微信渠道。 |
| `serverchan3` | Server酱³ | 新版 Server酱通道。 |

推荐设置顺序：添加通知渠道 → 返回通知通用选项 → 启用通知 → `validate-config` → 执行一次 `run` 验证实际送达。

#### 企业微信自建应用 `wecom`

准备步骤：

1. 登录企业微信管理后台。
2. 在企业信息中复制企业 ID。
3. 在“应用管理”中创建或打开自建应用。
4. 复制应用的 AgentId 和 Secret。
5. 将接收成员加入应用可见范围。
6. 如果后台要求可信 IP，将运行程序机器的公网出口 IP 加入白名单。

```yaml
- type: wecom
  corp_id: "${WECOM_CORP_ID}"
  agent_id: "${WECOM_AGENT_ID}"
  secret: "${WECOM_SECRET}"
  to_user: "@all"
  api_url: null
```

| 字段 | 中文说明 |
|---|---|
| `corp_id` | 企业 ID，必填。 |
| `agent_id` | 自建应用 AgentId，必填，建议加引号。 |
| `secret` | 自建应用 Secret，必填且敏感。 |
| `to_user` | 接收成员的企业微信 UserID，不是昵称或手机号；默认 `@all`。多人使用 `UserID1|UserID2`。 |
| `api_url` | 企业微信 API 根地址；`null` 使用 `https://qyapi.weixin.qq.com/`，普通用户无需修改。 |

`@all` 只影响应用可见范围内的成员。提示“成员不存在”时检查 UserID 与应用可见范围；获取 Token 失败时检查企业 ID、Secret 和可信 IP。

#### 企业微信群机器人 `wecomrobot`

1. 打开目标企业微信群设置。
2. 添加群机器人。
3. 复制完整 Webhook 地址。
4. 如需在消息中 `@` 某个成员，准备该成员绑定的完整手机号。

```yaml
- type: wecomrobot
  url: "${WECOM_ROBOT_URL}"
  mobile: null
```

| 字段 | 中文说明 |
|---|---|
| `url` | 群机器人完整 Webhook URL，必填且敏感。 |
| `mobile` | 可选手机号，用于在群消息中 `@` 该成员；不需要时为 `null`。 |

机器人只会向创建它的群发送消息。`mobile` 不是单独接收目标，消息仍发送到整个群。

#### WxPusher `wxpusher`

1. 在 WxPusher 控制台创建应用。
2. 复制应用 AppToken。
3. 个人推送：让用户关注应用并取得 UID。
4. 主题推送：创建主题并取得数字 Topic ID。

```yaml
- type: wxpusher
  app_token: "${WXPUSHER_APP_TOKEN}"
  uids:
    - "UID_xxx"
  topic_ids: []
  api_url: null
```

`uids` 与 `topic_ids` 至少有一项非空，也可以同时使用。菜单中多个 UID 或 Topic ID 使用英文逗号分隔。`api_url: null` 使用官方完整发送接口；自定义时必须填写完整发送接口 URL。

#### PushPlus `pushplus`

1. 登录 PushPlus 并绑定微信。
2. 复制个人 Token。
3. 个人推送时 `topic` 为 `null`；群组推送时填写群组编码。

```yaml
- type: pushplus
  token: "${PUSHPLUS_TOKEN}"
  topic: null
```

`topic` 是群组编码，不是通知标题。程序使用固定官方接口，当前没有独立 `api_url` 或代理字段。

#### Server酱 Turbo `ftqq`

```yaml
- type: ftqq
  sendkey: "${FTQQ_SENDKEY}"
  api_url: null
```

登录 Server酱 Turbo、绑定微信通道并复制 SendKey。`api_url: null` 使用官方地址；自定义值应是 HTTP/HTTPS API 根地址。

#### Server酱³ `serverchan3`

```yaml
- type: serverchan3
  sendkey: "${SERVERCHAN3_SENDKEY}"
  tags: "mihoyo|签到"
```

Server酱³ SendKey 必须类似 `sctp12345t...`，不能与 Server酱 Turbo 的 SendKey 混用。`tags` 可选，多个标签按服务商要求使用 `|` 分隔。

### 12.3 SMTP 邮件详细设置

先在邮件服务商后台开启 SMTP，并创建独立授权码；不要直接使用邮箱登录密码。常见配置如下：

```yaml
- type: smtp
  host: "smtp.example.com"
  port: 465
  from: "sender@example.com"
  to: "receiver@example.com"
  username: "${SMTP_USERNAME}"
  password: "${SMTP_PASSWORD}"
  subject: "MihoyoBBSTools RS"
  tls: implicit
  timeout_seconds: null
```

字段说明：

- `host`：邮件服务商提供的 SMTP 主机名，不要填写 `https://`。
- `port`：SMTP 端口。隐式 TLS 通常为 `465`，STARTTLS 通常为 `587`，以服务商说明为准。
- `from`：发件人邮箱地址，通常必须与 SMTP 账号一致。
- `to`：单个收件人邮箱地址。
- `username`：SMTP 登录用户名，通常是完整邮箱地址。
- `password`：SMTP 授权码或专用密码，建议通过环境变量提供。
- `subject`：固定邮件主题；邮件正文会包含本次执行状态和完整任务报告。
- `tls`：`implicit` 为连接建立时直接加密，`starttls` 为连接后强制升级加密，`none` 为不加密。
- `timeout_seconds`：可选 SMTP 连接与发送超时；`null` 复用 `runtime.request_timeout_seconds`，范围为 1 到 300 秒。

推荐优先使用 `implicit` 或 `starttls`。`none` 会明文传输用户名和授权码，只能用于可信隔离网络，不应连接公网邮件服务器。发送失败只记录为该通知渠道失败，不改变核心签到结果。

### 12.4 Windows 本地通知详细设置

```yaml
- type: windows_toast
  title_prefix: "MihoyoBBSTools RS"
```

该渠道使用 Windows 自带的 WinRT Toast，通过系统 `powershell.exe` 提交通知，不需要安装 Python 包或其他运行库。`title_prefix` 会显示在任务状态标题之前；设为 `""` 时只显示任务状态标题。通知正文最多提交 4096 个字符，超出部分自动截断。

使用限制：

- 只支持 Windows；在 Linux 配置该渠道会得到明确的不支持错误。
- 必须有已登录的交互桌面会话。Windows 服务和 Session 0 会明确报告通知提交失败。
- 任务计划程序应选择“仅当用户登录时运行”；选择“无论用户是否登录都运行”时，命令可能成功但桌面不会显示通知。
- 程序会隐藏 PowerShell 窗口，并在 15 秒内未完成时终止通知子进程。
- 标题和正文通过子进程环境变量传递，不会拼接为 PowerShell 代码。

### 13.4.1 自动刷新与完成确认

- 国内签到或米游社任务明确返回认证失效时，程序会使用账号 SToken 刷新 `cookie_token`，每个任务流程最多一次，然后重试。
- 普通 YAML Cookie 会安全写回配置。Cookie 使用 `${ENV_NAME}` 时只更新本次运行，不会把环境变量中的 Secret 写入 YAML；请随后更新环境变量或 GitHub Secret。
- HoYoLAB 使用国际服凭据，不使用这套国内 SToken 刷新。
- 国内和 HoYoLAB 签到提交后会再次查询服务端。复查仍未签到时，只重试该角色，最多执行 `runtime.task_max_attempts` 次；确认成功后显示尝试次数、累计签到天数及今日奖励，奖励详情获取失败不影响已确认的成功状态。
- 社区签到、阅读、点赞和分享仅在配置启用且服务端返回对应的已映射任务（`58`–`61`）时执行。每轮动作后等待 3 秒，只复查对应任务；明确未完成时按剩余进度进入下一轮，确认完成后立即停止该任务。多个任务独立计数，未知任务只记录、不执行。
- 任何状态复查请求失败都会停止对应副作用动作，不把“状态未知”当作“未完成”；达到 `runtime.task_max_attempts` 后仍明确未完成时报告“状态同步超时”。

默认日志配置：

```yaml
runtime:
  log_level: info
  logging:
    enabled: true
    directory: logs
    file_prefix: mihoyo-bbs-tools
```

默认生成：

```text
logs/mihoyo-bbs-tools_2026-07-13.log
```

日期变化后会写入新的日期文件。旧配置中的 `file_prefix: mihoyo-bbs-tools.log` 仍兼容，也会生成新的下划线日期格式。

日志级别：`trace`、`debug`、`info`、`warn`、`error`。日志可能包含账号备注、角色 UID 尾号和执行结果，但不会主动输出 Cookie、SToken、通知 Token 或代理认证信息。

## 15. 自动运行

建议先手动执行 `validate-config` 和 `run`，确认成功后再创建计划任务。

Windows 用户推荐直接使用：

```powershell
.\MihoyoBBSToolsRS.exe automation install --time 09:00
.\MihoyoBBSToolsRS.exe automation status
.\MihoyoBBSToolsRS.exe automation run-now
.\MihoyoBBSToolsRS.exe automation uninstall
```

`install` 会显示固定任务名、程序绝对路径、工作目录、配置路径、每日本地时间和失败重试次数，然后创建或更新同名任务，不会重复创建多个任务。默认只在用户登录时运行；如需未登录时也运行，可增加 `--run-whether-logged-on`，但 Windows 本地桌面通知届时不可见。`--enable-windows-notification` 可同时启用本地通知渠道。

`status` 显示启用状态、下次/上次运行、上次退出码，并检测程序和配置路径是否失效。`uninstall` 只删除固定名称的任务，不删除配置和日志。

手工创建 Windows 任务计划程序时可设置：

```text
程序：D:\tools\MihoyoBBSToolsRS.exe
参数：run --config D:\tools\config\config.yaml
起始于：D:\tools
```

Linux cron 示例：

```cron
15 8 * * * cd /opt/mihoyo-bbs-tools && ./MihoyoBBSToolsRS run >> /dev/null 2>&1
```

避免同一配置被多个任务同时运行，以免重复请求或同时修改配置。

### Kubernetes CronJob

仓库的 `deploy/kubernetes/cronjob.yaml` 默认使用 `Asia/Shanghai` 时区，每天 `00:05` 执行一次。先准备完整配置，再以 Secret 创建或更新，不要用 ConfigMap 保存 Cookie 和通知凭据：

```bash
kubectl create secret generic mihoyo-bbs-tools-config \
  --from-file=config.yaml=./config/config.yaml \
  --dry-run=client -o yaml | kubectl apply -f -
kubectl apply -f deploy/kubernetes/cronjob.yaml
```

生产环境应把清单中的 `image` 从 `latest` 改为已发布的固定版本标签或镜像摘要。配置卷不可写，清单已经使用 `--read-only`；凭据自动刷新只对当次运行有效，需要持久化时应在集群外安全更新 Secret。不要在 `config.yaml` 中同时启用 `runtime.schedule`，否则会在单次 Job 内再次启动常驻调度。

查看定时任务和最近一次日志：

```bash
kubectl get cronjob mihoyo-bbs-tools-rs
kubectl get jobs --selector=app.kubernetes.io/name=mihoyo-bbs-tools-rs
kubectl logs job/<任务名称>
```

手动验证一次：

```bash
kubectl create job --from=cronjob/mihoyo-bbs-tools-rs mihoyo-bbs-tools-manual
kubectl logs -f job/mihoyo-bbs-tools-manual
```

清单的 `timeZone` 需要 Kubernetes 1.27 或更高版本。旧集群应删除该字段，并根据控制平面时区换算 Cron；UTC 集群可使用 `5 16 * * *`。文件日志位于临时 `emptyDir`，Job 删除后不会保留，长期日志应由集群日志系统收集。

### Nix 与 NixOS

已启用 Flakes 的 Linux 可以直接从仓库运行：

```bash
nix run github:bcmdy/mihoyo-bbs-tools-rs -- run
```

在源码目录构建和运行：

```bash
nix build
./result/bin/MihoyoBBSToolsRS run
```

当前 Flake 声明 `x86_64-linux` 和 `aarch64-linux`。配置仍从当前工作目录的 `config/config.yaml` 读取；Nix store 是只读的，不要把真实配置放进源码或构建输出。包会把完整模板安装到 `$out/share/mihoyo-bbs-tools/`，并把本使用说明安装到 `$out/share/doc/mihoyo-bbs-tools/`。

Nix 包显示的版本取自 `Cargo.toml`，GitHub Release 压缩包和普通 Actions 构建仍使用标签或工作流输入的动态版本。首次引入或更新 `nixpkgs` 时，维护者应通过远程“生成依赖锁文件”工作流更新 `flake.lock`，不要手写锁文件。

### 容器镜像与 ARM 平台

版本标签会发布以下多架构镜像：

```bash
docker pull ghcr.io/bcmdy/mihoyo-bbs-tools-rs:<版本>
```

同一标签包含 `linux/amd64`、`linux/arm64` 和 `linux/arm/v7`，Docker 或 Kubernetes 会按节点架构选择镜像。稳定版本标签同时更新 `latest`；`main` 分支只发布 `main` 和 `sha-<提交>`，不会覆盖稳定版。

运行时只读挂载配置，不要把真实配置复制进镜像：

```bash
docker run --rm \
  -v "$PWD/config:/app/config:ro" \
  ghcr.io/bcmdy/mihoyo-bbs-tools-rs:<版本> \
  run --config /app/config/config.yaml --read-only
```

GitHub Release 还提供独立的 Linux `x86_64`、`arm64`、`armv7` 压缩包和 Windows `x86_64` 压缩包。Linux ARM 文件使用 GNU libc；选择独立压缩包时，先用 `uname -m` 确认设备架构，并核对同名 `.sha256` 文件。

## 16. 旧配置迁移

位置参数方式：

```powershell
.\MihoyoBBSToolsRS.exe migrate-config old.yaml new.yaml
```

兼容参数方式：

```powershell
.\MihoyoBBSToolsRS.exe migrate-config --input old.yaml --output new.yaml
```

省略输出文件时，会在源文件同目录创建 `.migrated.yaml` 或 `.migrated.yml` 文件。程序不会覆盖已有目标文件。迁移结果可能包含真实凭据，应妥善保存。

旧版 Python v11–v15 YAML 可以直接运行，但只会在内存中转换。程序若自动刷新 `cookie_token`，本轮会使用新凭据重试，不会把新版 `accounts` 结构误写回旧文件；后续运行仍会读取旧凭据。长期使用前应先执行 `migrate-config`，再改用生成的新版 YAML。

## 17. 运行状态与退出码

默认文本模式显示中文状态；JSON 与日志接口继续使用下列稳定英文枚举：

- `Success`：任务成功。
- `AlreadyCompleted`：今日已经完成。
- `Skipped`：任务被关闭、没有绑定角色或无需执行。
- `StateSyncTimeout`：动作已执行，但达到业务复查上限后服务端仍未确认对应任务完成。
- `AuthenticationFailed`：Cookie 或凭据失效。
- `CaptchaRequired`：需要验证码。
- `NetworkFailed`：请求超时、连接失败或代理故障。
- `Failed`：其他业务失败。

主要退出码：

| 退出码 | 含义 |
|---|---|
| `0` | 成功、已完成或仅包含跳过项 |
| `1` | 普通任务失败 |
| `2` | 配置错误 |
| `3` | 认证失败 |
| `4` | 需要验证码 |
| `5` | 网络或代理失败 |

## 18. 常见问题

### YAML 报 `did not find expected key`

通常是缩进、冒号、引号或列表结构错误。重点检查报错行前几行。不要在 `providers: []` 后直接追加渠道。

### 环境变量未设置

配置中的 `${NAME}` 会在读取时展开。即使对应通知暂时关闭，只要占位符仍存在，也必须设置环境变量或改成实际值/null。

### 无法获取米游社昵称

确认 Cookie 完整且未过期，Cookie 中能识别 UID，并检查当前网络能否访问米游社。昵称查询使用直接网络连接，不读取尚未添加账号的代理设置；已有账号可在 `config setup → 账号 → 更新 Cookie` 中重试。

### Cookie 中缺少 SToken

重新获取包含 SToken 的完整 Cookie。只包含游戏签到字段的 Cookie 不足以添加完整账号。

### 显示“没有绑定角色”

该账号没有绑定对应游戏角色，或绑定区域不匹配。可以在设置菜单中关闭该游戏。

### Telegram 推送失败

检查 Bot Token、Chat ID、`api_url` 和 Telegram 独立代理。国内网络通常需要可访问 Telegram API 的代理。

### Windows 本地通知没有显示

确认程序在 Windows 的已登录桌面会话中运行，并将任务计划程序设置为“仅当用户登录时运行”。远程 CI、Windows 服务、锁屏前创建的 Session 0 和无用户登录的计划任务不具备可显示通知的桌面；这种情况下应改用 Telegram、邮件或其他网络通知渠道。

### 日志没有生成

检查 `runtime.logging.enabled`、日志目录写入权限以及运行时工作目录。

## 19. 升级与备份

交互式配置保存前会在 `config/backups/` 自动保留最近五份。也可以手动管理：

```powershell
.\MihoyoBBSToolsRS.exe config backup
.\MihoyoBBSToolsRS.exe config list-backups
.\MihoyoBBSToolsRS.exe config restore config\backups\config-20260716-153000.yaml
```

列表只显示时间、配置版本和账号数量。恢复只接受当前配置 `backups` 目录内的文件，恢复前先校验备份并再次备份当前配置；非交互环境必须显式增加 `--yes`。

检查新版本：

```powershell
.\MihoyoBBSToolsRS.exe update check
```

该命令只读取 GitHub 最新稳定 Release 信息，不会自动下载或替换程序。升级前仍建议额外备份：

```text
config/config.yaml
logs/
```

替换可执行文件后先运行：

```powershell
.\MihoyoBBSToolsRS.exe version
.\MihoyoBBSToolsRS.exe validate-config
```

不要直接用新模板覆盖真实配置。需要新增字段时，可对照发布包中的 `config/config.example.yaml`，或使用 `config setup` 修改。

## 20. 安全与泄露处理

Cookie、SToken、云游戏 Token、通知 Token、Webhook、SMTP 密码、含认证信息的代理地址、真实配置和未经检查的日志都应按密码文件保护。

- 个人电脑：把 `config/config.yaml` 放在非共享目录，只允许当前用户读取，不要提交到 Git。
- GitHub Actions：使用加密 Secrets；不要在 Pull Request、日志或 Artifact 中输出真实配置。
- Docker：运行时只读挂载配置，不要通过 `ARG`、`ENV` 或 `COPY` 写进镜像层。
- Kubernetes：使用 Secret 只读挂载，不要用 ConfigMap 保存 Cookie；只读卷中的凭据刷新不会持久化。
- 上层程序或云函数：通过标准输入配合 `--config - --read-only` 注入；需要禁止通知时同时添加 `--no-notify`。

`${ENV_NAME}` 提供的凭据和只读配置只在本轮内存中刷新，不会把展开后的 Secret 写回 YAML。日志虽然会主动隐藏 Cookie、Token 和密码，但仍可能包含账号名称、角色 UID 尾号、请求目标和任务结果；反馈问题前应人工检查并脱敏，不要直接上传整个 `logs` 目录。

HTTPS 默认校验证书且不能全局关闭。SMTP 推荐使用 `implicit` 或 `starttls`；`tls: none` 只适用于可信隔离网络。验证码端点、自定义通知 API 和代理服务都可能接触敏感数据，使用前应确认服务可信。

如果凭据已经出现在聊天、截图、日志、Artifact 或 Git 历史中，应视为已经泄露：立即撤销或轮换对应 Cookie、Token、Webhook、SMTP 授权码和代理密码，停止使用旧凭据的计划任务或容器，清理可下载产物，再更新安全存储后重新启用。仅删除文件或提交不能使 Git 历史中的 Secret 失效。
