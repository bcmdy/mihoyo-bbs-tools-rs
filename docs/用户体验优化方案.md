# 用户体验优化方案

> 状态：待实施
>
> 本文面向项目维护者，记录从普通用户视角出发的体验问题、建议交互、实施边界和验收标准。用户实际使用方法仍以 [快速开始](快速开始.md) 与 [使用说明](使用说明.md) 为准。

## 1. 背景

项目已经具备国内游戏签到、HoYoLAB、米游社任务、云游戏、多账号、代理、验证码、通知、日志、批量运行和多种部署方式。现阶段主要问题不是功能覆盖不足，而是普通用户仍需理解较多命令、配置层级和技术术语，才能完成首次配置、长期自动运行以及失败后的恢复。

当前典型路径为：

1. 下载并解压发布包。
2. 在程序目录打开终端。
3. 自行取得包含 UID、SToken 和可选 MID 的完整 Cookie。
4. 执行 `config add-account`。
5. 按需进入 `config setup` 设置任务和通知。
6. 执行 `validate-config`。
7. 执行 `run`，再根据逐项报告判断是否成功。
8. 另行配置 Windows 任务计划程序、cron 或常驻 `schedule`。

这条路径对熟悉命令行和 YAML 的用户足够明确，但对普通 Windows 用户仍存在以下阻力：

- 不容易判断 Cookie 是否完整，失败后通常只能重新获取并重试。
- Cookie、通知 Token 和 SMTP 密码等输入虽然不会写入日志，但会显示在终端中。
- 数字菜单没有持续显示当前启用状态，重新选择时容易误关闭已有任务。
- 菜单中的 `true/false`、`null`、字段名和通知渠道标识偏向配置实现，而不是用户语义。
- 配置修改即时写入，没有统一的变更摘要、确认、撤销和自动备份入口。
- `validate-config` 只验证本地配置，`run` 又会执行真实任务，中间缺少只读诊断能力。
- 通知渠道没有独立测试命令，只能通过完整运行间接验证。
- 文本报告逐条输出英文状态枚举，缺少汇总、重点失败和修复建议。
- Windows 自动运行仍需用户手工操作任务计划程序，和“自动签到”的核心预期存在距离。

## 2. 目标

本方案希望达到以下结果：

- 普通用户无需编辑 YAML，即可完成首个账号、任务选择、通知测试和首次运行。
- 交互式终端中不显示 Cookie、Token、密码和含认证信息的代理地址。
- 用户修改任务或游戏时，始终能看到当前状态和待保存变化。
- 一次诊断可以区分配置、凭据、网络、代理、验证码和通知问题。
- 默认文本报告先给结论，再给失败详情；自动化 JSON 契约保持稳定。
- Windows 用户可以通过程序安装、查看和移除每日定时任务。
- 所有配置写入都具备明确确认、原子替换和可恢复能力。

建议使用以下可观测指标验证效果：

- 首次配置在 5 分钟内完成，且不要求打开 YAML。
- 从解压到首次成功运行所需的必输命令不超过 2 条。
- 交互式输入过程中，终端和日志均不出现完整敏感值。
- 常见失败均能在报告中给出下一步动作或对应菜单路径。
- 现有脚本、旧配置和 JSON 输出不因体验改造而失效。

## 3. 非目标

本轮不以增加新的签到平台、通知渠道或部署平台为目标，也不立即开发完整桌面 GUI。

不建议自动读取浏览器 Cookie。该行为会扩大本地权限和浏览器兼容范围，也更容易造成用户误解。首期应提供安全输入、字段检查和明确获取说明；只有在安全模型、平台规则和维护成本都可接受时，才单独评估浏览器集成。

## 4. 设计原则

### 4.1 默认面向普通用户

默认文本、菜单和错误提示使用用户能够直接理解的中文。字段名、枚举值、退出码和 JSON 结构继续为高级用户与自动化保留。

### 4.2 交互与自动化分离

- TTY 中允许向导、确认和隐藏输入。
- 非 TTY 环境不得等待交互输入。
- 自动化继续使用显式参数、标准输入和稳定 JSON。
- `run`、`checkin`、`schedule` 不得因为缺少配置而自动进入向导。

### 4.3 敏感信息最小暴露

- 不回显 Cookie、SToken、通知 Token、密码和认证代理。
- 不把敏感值放入命令参数、日志、错误、诊断报告或崩溃信息。
- 测试通知和在线诊断只输出渠道类型、账号备注和脱敏标识。
- 新依赖必须经过安全、维护状态和体积评估。

### 4.4 先确认，再写入

- 菜单操作先修改暂存状态。
- 保存前展示不含敏感值的变更摘要。
- 用户明确确认后再执行校验和原子替换。
- 删除账号、删除通知渠道和恢复备份必须二次确认。

### 4.5 保持兼容

- 现有命令及默认配置路径保持可用。
- 初期不提升配置 `version`。
- `--output json` 的字段和英文枚举保持不变；需要调整时新增 schema 版本。
- 旧版配置迁移与青龙、DaCapo、目录批量运行不依赖交互能力。

## 5. P0：敏感输入不回显

### 5.1 用户行为

在交互式终端中输入以下内容时，屏幕不显示字符：

- 国内 Cookie 和 HoYoLAB Cookie。
- 云游戏 Token。
- 通知 Token、Secret、SendKey、Webhook、SMTP 密码等凭据。
- 包含用户名或密码的代理 URL。

输入完成后只显示“已读取”或脱敏结果，例如：

```text
请输入完整 Cookie（输入内容不会显示）：
已识别 UID：****1234，SToken：已包含，MID：已包含
```

### 5.2 实现建议

新增统一的终端输入层，例如 `src/config/input.rs`：

- `prompt_text`：普通文本输入。
- `prompt_secret`：TTY 下关闭回显，非 TTY 下读取一行。
- `prompt_optional_secret`：支持留空保留和显式清空。
- `confirm`：统一处理 yes/no、EOF 和默认值。

可以评估使用维护活跃的隐藏输入依赖，也可以封装 Windows 与 Unix 终端 API。无论选择哪种实现，调用方不得再次把返回值写入格式化字符串。

`ProviderField` 应增加 `secret` 元数据，不能只根据字段名临时判断。至少以下字段应标记为敏感：

- `token`、`bot_token`、`app_token`、`sendkey`、`secret`、`key`、`password`。
- 完整 Webhook、可能携带凭据的 `url`、`api_url` 和 `proxy`。
- SMTP `username` 可按普通文本处理，但不得在日志中主动输出。

### 5.3 兼容规则

- `config add-account < cookie.txt` 继续可用，但不输出交互提示。
- 标准输入为空时返回明确错误，不能创建空配置。
- Windows Terminal、PowerShell、CMD 和常见 Linux 终端至少各验证一种。
- EOF、Ctrl+C 和终端读取失败不写配置。

### 5.4 验收标准

- TTY 测试确认屏幕不出现输入值。
- 日志、错误和 Debug 输出不包含测试 Secret。
- 非 TTY 测试确认不会挂起。
- 添加账号和更新 Cookie 使用同一套隐藏输入能力。

## 6. P0：配置状态可见、修改可确认

### 6.1 菜单展示

任务、游戏、通知和账号开关应显示当前状态，而不是只显示编号：

```text
账号：mys用户:示例（小号）

1. [x] 国内游戏签到
2. [ ] HoYoLAB 签到
3. [x] 米游社任务
4. [ ] 国内云游戏
5. [ ] 海外云游戏
6. [ ] Web 活动

输入编号切换，a 全选，n 全不选，s 保存，0 返回：
```

布尔设置使用“启用/关闭”，不要求普通用户输入 `true/false`。仍可接受 `y/n` 作为快捷输入。

通知渠道列表应使用中文名称和简短用途，例如“Telegram 机器人”“Windows 本地通知”“企业微信群机器人”，技术标识放在括号中。

### 6.2 暂存与确认

建议引入配置编辑会话：

1. 读取原始 YAML 和已解析配置。
2. 在内存中的 YAML 副本上应用菜单修改。
3. 每一步进行局部校验，不立即替换真实文件。
4. 退出设置时展示变更摘要。
5. 用户确认后执行完整校验和原子替换。

摘要示例：

```text
待保存变更：
- 账号“小号”：启用星穹铁道签到
- 账号“小号”：关闭米游社点赞
- 通知：新增 Windows 本地通知

保存这些变更？[Y/n]
```

摘要不得显示 Secret 的旧值或新值，只显示“已新增”“已更新”“已清空”。

### 6.3 删除保护

- 删除账号前显示账号名、备注和受影响任务。
- 删除通知渠道前显示渠道类型和脱敏接收目标。
- 需要再次输入 `yes` 或账号编号确认。
- 取消后不得写文件。

### 6.4 验收标准

- 所有可切换列表都显示当前状态。
- 返回上级菜单不会丢失暂存变化。
- 未确认、EOF 或 Ctrl+C 不修改真实配置。
- 保存失败时原配置保持不变。
- 未知 YAML 字段和字段顺序策略继续按现有编辑器约定保留。

## 7. P0：面向用户的运行报告

### 7.1 默认文本输出

默认文本模式先输出汇总，再输出需要关注的详情：

```text
运行完成：成功 5，今日已完成 3，跳过 2，失败 1

需要处理：
[认证失效] 小号 / 原神：Cookie 已失效
处理方式：运行 `config setup`，选择“账号 -> 更新 Cookie”

详细成功项可使用 `run --verbose` 查看。
```

建议将文本状态本地化：

| 内部状态 | 默认文本 |
|---|---|
| `Success` | 成功 |
| `AlreadyCompleted` | 今日已完成 |
| `Skipped` | 已跳过 |
| `AuthenticationFailed` | 认证失效 |
| `CaptchaRequired` | 需要验证码 |
| `NetworkFailed` | 网络失败 |
| `Failed` | 执行失败 |

默认可折叠大量成功和跳过项，`--verbose` 输出全部记录。是否折叠应在发布说明中明确，避免用户误以为任务未执行。

### 7.2 修复建议

为常见失败建立稳定的“错误类别 -> 建议动作”映射：

- 认证失效：更新对应账号 Cookie。
- 缺少 SToken/MID：重新获取完整 Cookie，并指出缺失字段。
- 需要验证码：配置验证码端点，或稍后重试。
- 代理失败：运行在线诊断，并检查对应账号代理。
- 通知失败：运行通知测试，不改变核心任务结果。
- 配置字段错误：显示字段路径、当前值类型和允许范围。

远端接口原始消息可作为详细信息保留，但不能成为用户看到的唯一解释。

### 7.3 自动化兼容

- JSON 中的 `schema_version`、`exit_code`、`records` 和 `notifications` 保持不变。
- JSON 枚举继续使用 snake_case 英文值。
- 日志继续进入标准错误或日志文件，JSON 标准输出只包含单个对象。

### 7.4 验收标准

- 每次文本运行都有汇总，即使记录为空。
- 非零退出时至少显示一个首要问题和建议动作。
- 多账号报告能够按账号归组或明确标识账号。
- 通知使用与终端相同的中文汇总，但不包含终端命令中的绝对敏感路径。

## 8. P1：一体化首次配置向导

### 8.1 命令入口

建议新增：

```text
MihoyoBBSToolsRS config init
```

`config init` 只负责首次创建。已有配置存在时默认拒绝覆盖，并提示改用 `config setup`。如未来需要重新初始化，应要求显式目标路径，不能提供简单的 `--force` 覆盖真实配置。

### 8.2 向导步骤

1. 检查系统、程序目录、配置路径和写入权限。
2. 隐藏输入国内 Cookie。
3. 本地解析并显示 UID、SToken、MID 的脱敏检查结果。
4. 在线获取米游社昵称和已绑定角色。
5. 选择需要启用的游戏和社区任务。
6. 询问是否添加第二个账号。
7. 可选添加一个通知渠道并立即发送测试消息。
8. 展示最终摘要并创建配置。
9. 询问是否执行首次运行。
10. Windows 下询问是否安装每日定时任务。

任何一步失败都不得留下半成品配置。用户主动跳过通知或自动运行不应产生警告。

### 8.3 Cookie 反馈

错误应尽量具体：

```text
无法添加账号：Cookie 中已识别 UID，但缺少 SToken。
请重新获取完整米游社 Cookie；只包含游戏签到字段的 Cookie 不足以执行社区任务。
```

不建议在向导中展示或保存 Cookie 原文，也不建议将 Cookie 字段数量作为“安全程度”指标。

### 8.4 验收标准

- 新用户无需打开 YAML 即可完成首个账号和任务选择。
- 向导中断时不产生 `config.yaml` 或临时文件。
- 已有配置不会被覆盖。
- 向导创建的配置可立即通过 `validate-config`。

## 9. P1：诊断与独立测试

### 9.1 `doctor` 命令

建议新增：

```text
MihoyoBBSToolsRS doctor
MihoyoBBSToolsRS doctor --online
MihoyoBBSToolsRS doctor --output json
```

默认离线检查：

- 配置文件是否存在、可读、版本受支持且结构有效。
- 配置目录和日志目录是否可写。
- 环境变量占位符是否完整。
- 代理 URL、验证码 URL 和通知 URL 是否可解析。
- 账号、任务和游戏配置是否存在明显冲突。
- 当前系统是否支持已配置的本地通知。

`--online` 额外检查：

- 米游社与 HoYoLAB 基础网络连通性。
- 各账号代理连通性。
- 凭据是否能够完成只读身份查询。
- 验证码端点是否能够访问；不主动触发真实验证码。
- 自定义通知 API 的基础地址是否可连接；不默认发送消息。

每项结果使用“通过/警告/失败/跳过”，并明确在线检查可能发起的请求。诊断不得执行签到、阅读、点赞、分享或云游戏领取。

### 9.2 通知测试

建议在现有 `config` 或独立命令组下增加：

```text
MihoyoBBSToolsRS notification list
MihoyoBBSToolsRS notification test
MihoyoBBSToolsRS notification test --provider 1
```

测试消息应明确标注“测试通知”，包含程序版本、渠道类型和发送时间，不包含账号任务详情。单个渠道失败不阻止继续测试其他渠道，最终返回聚合状态。

### 9.3 验收标准

- 离线 `doctor` 不访问网络。
- 在线诊断不产生任何任务副作用。
- 每个失败项包含原因、影响和建议动作。
- JSON 诊断输出具有独立 schema 版本。
- 通知测试可以精确选择渠道，并正确区分配置错误、网络失败和服务端拒绝。

## 10. P1：自动运行管理

### 10.1 命令设计

为避免改变现有常驻 `schedule` 命令语义，建议新增独立命令组：

```text
MihoyoBBSToolsRS automation install
MihoyoBBSToolsRS automation status
MihoyoBBSToolsRS automation run-now
MihoyoBBSToolsRS automation uninstall
```

首期优先支持 Windows 任务计划程序。Linux 首期可以输出经过校验的 cron 或 systemd user timer 模板，后续再评估自动安装。

### 10.2 Windows 行为

`automation install` 应允许选择：

- 每日执行时间和时区提示。
- 使用的配置文件。
- 是否只在用户登录时运行。
- 是否启用 Windows 本地通知。
- 失败后的重试次数。

安装前显示将创建的任务名称、程序绝对路径、工作目录和参数。程序移动后，`automation status` 应检测路径失效，并提示重新安装。

`automation status` 至少显示：

- 是否已安装和是否启用。
- 下次运行时间。
- 上次运行时间和退出码。
- 当前程序及配置路径是否存在。

### 10.3 安全与兼容

- 不把 Cookie 或其他 Secret 写入任务参数。
- 任务只引用受控配置路径。
- 安装和删除只操作本项目固定名称或带明确实例 ID 的任务。
- 不递归删除目录，不自动删除用户配置和日志。
- `create-launcher` 保留，作为手动双击入口。

### 10.4 验收标准

- 普通 Windows 用户无需打开任务计划程序即可完成安装。
- `status` 能检测程序移动、配置缺失和任务被禁用。
- 重复安装不会创建多个同名任务。
- 卸载后配置和日志保持不变。

## 11. P2：备份、升级和长期维护

### 11.1 配置备份与恢复

每次交互式写入前，在受控目录创建权限受限的备份，例如：

```text
config/backups/config-20260716-153000.yaml
```

建议提供：

```text
MihoyoBBSToolsRS config backup
MihoyoBBSToolsRS config list-backups
MihoyoBBSToolsRS config restore <BACKUP>
```

规则：

- 默认保留最近 5 份，可配置但应有上限。
- 备份与原配置使用相同的安全权限。
- 恢复前先校验备份并备份当前文件。
- 列表只显示时间、版本、账号数量等脱敏信息。
- 自动化凭据刷新写回可以单独评估是否触发备份，避免每日产生大量文件。

### 11.2 新版本提示

首期只做可选检查，不自动下载或替换程序：

```text
MihoyoBBSToolsRS update check
```

检查结果应显示当前版本、最新稳定版本、发布页面和配置兼容提示。普通 `run` 中的自动检查默认关闭，或最多按较长周期执行，并允许彻底关闭。

### 11.3 Cookie 与故障说明

补充独立文档，说明：

- 工具需要哪些 Cookie 字段以及各自用于什么功能。
- 如何判断复制内容是否完整。
- 为什么不应通过聊天、截图、命令参数或公开仓库传递 Cookie。
- 常见“有 UID 但没有 SToken”“V2 SToken 缺少 MID”的处理方式。
- 不提供来源不明的扩展程序和脚本下载链接。

### 11.4 本地界面评估

如果后续用户反馈仍集中在终端门槛，可评估 TUI 或仅监听 `127.0.0.1` 的本地 Web 配置界面。评估前必须先明确：

- CSRF、跨站访问和本地端口暴露风险。
- Secret 在浏览器 DOM、剪贴板和开发者工具中的暴露范围。
- Windows 防火墙提示和端口冲突处理。
- 是否能够复用同一套配置编辑、校验和诊断 API。

在配置编辑 API 尚未稳定前，不建议直接开发 GUI。

## 12. 建议命令总览

| 命令 | 用途 | 优先级 |
|---|---|---|
| `config init` | 一体化首次配置 | P1 |
| `config setup` | 显示当前状态并暂存修改 | P0 |
| `doctor` | 离线环境与配置诊断 | P1 |
| `doctor --online` | 只读网络与凭据诊断 | P1 |
| `notification list/test` | 独立测试通知渠道 | P1 |
| `run --verbose` | 展开全部任务记录 | P0 |
| `automation install/status/run-now/uninstall` | 管理自动运行 | P1 |
| `config backup/list-backups/restore` | 配置恢复 | P2 |
| `update check` | 检查稳定版本 | P2 |

命令名称在实现前应使用 Clap 原型验证帮助文本和冲突关系。若最终命名变化，必须同步更新 README、快速开始、完整使用说明和 Shell 示例。

## 13. 文件级实施建议

### `src/config/interactive.rs`

- 替换分散的输入函数，接入统一终端输入层。
- 菜单显示当前状态、中文渠道名称和快捷操作。
- 增加暂存、变更摘要、确认与删除保护。

### `src/config/editor.rs`

- 添加账号、更新 Cookie 和通知 Secret 统一使用隐藏输入。

- 提供诊断所需的只读配置检查结果。
- 将用户可修复的校验错误映射为稳定类别。
- 保持现有配置版本和反序列化兼容。

### `src/cli.rs`

- 增加 `config init`、`doctor`、`notification`、`automation` 等命令模型。
- 为文本运行增加 `--verbose`。

### `src/main.rs`

- 分派新增命令。
- 确保诊断和通知测试使用正确退出码。

### `src/service/report.rs`

- 增加计数汇总和中文文本渲染。

### `src/push/`

- 提供单渠道测试入口。
- 复用现有发送实现，不复制渠道协议代码。
- 测试结果继续执行 Secret 脱敏。

### 新模块建议

- `src/config/input.rs`：统一普通、敏感和确认输入。
- `src/doctor.rs`：离线与在线诊断编排。
- `src/automation.rs`：平台自动运行管理；平台实现可继续拆分。

## 14. 测试计划

### 14.1 输入安全

- TTY 敏感输入不回显。
- 非 TTY 可以从标准输入读取，且不会输出原文。
- EOF、取消和读取失败不写配置。
- Cookie、Token、Webhook、代理密码不出现在日志和 Debug 输出。

### 14.2 配置交互

- 当前任务、游戏和通知状态显示正确。
- 多次切换后保存结果正确。
- 返回、取消、EOF 和拒绝确认不写文件。
- 删除账号和渠道必须确认。
- 暂存配置完整校验失败时原文件不变。
- 未知字段继续保留。

### 14.3 报告

- 各状态计数正确。
- 默认文本包含汇总和首要问题。
- `--verbose` 包含全部记录。
- 认证、验证码、网络和普通失败映射到正确建议。
- JSON 快照与当前 schema 保持兼容。

### 14.4 首次向导

- 新目录成功创建首个配置。
- 已有配置拒绝覆盖。
- 缺少 UID、SToken 或 MID 时给出对应提示。
- 中途失败或取消不残留文件。
- 创建结果可直接加载和运行。

### 14.5 诊断与通知

- 离线诊断不创建 HTTP 请求。
- 在线诊断只调用只读接口。
- 代理、DNS、TLS、超时和认证失败分类正确。
- 通知测试支持单渠道和全部渠道。
- 测试消息及错误不包含 Secret。

### 14.6 自动运行

- Windows 任务创建参数、工作目录和配置路径正确转义。
- 带空格、中文和 `%` 的路径正常工作。
- 重复安装、程序移动、配置缺失和任务禁用可被检测。
- 卸载只删除目标任务。
- 非 Windows 平台返回明确支持说明。

### 14.7 发布前验证

每个里程碑至少执行：

```text
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-targets
```

输入隐藏、Windows 任务计划程序、终端中文显示和 Ctrl+C 行为需要保留人工验收清单，不能只依赖单元测试。

## 15. 实施里程碑

### 里程碑一：安全与结果可读性

1. 统一敏感输入并关闭 TTY 回显。

## 16. 总体验收标准

全部核心改进完成后，应满足：

- 新用户可以只通过 `config init` 和 `run` 完成首次使用。
- 所有交互式 Secret 默认不回显。
- `config setup` 明确显示当前状态，退出前统一确认。
- `doctor` 能在不执行真实任务的前提下定位常见问题。
- 每个通知渠道可以独立测试。
- 默认文本报告包含中文汇总、重点失败和处理建议。
- Windows 自动运行可以由程序安装、检查和移除。
- 配置误改可以通过受控备份恢复。
- 现有配置、CLI 自动化和 JSON 消费者继续工作。
- README、快速开始、使用说明、配置参考和 `--help` 与实际行为一致。

实施顺序应优先解决“敏感信息暴露”和“用户不知道下一步做什么”，再扩展首次向导和系统自动化。新增通知渠道、部署方式或高级配置能力不应挤占上述核心体验工作的优先级。
