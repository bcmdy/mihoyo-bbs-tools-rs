# 报告与重试机制修改需求

## 1. 背景与目标

当前报告默认只展示汇总和失败项，游戏签到奖励、社区签到结果等成功详情通常只有在 `run --verbose` 中可见。同时，社区签到与米游币任务存在服务端状态延迟，验证码平台也可能返回临时失败结果，容易造成误报失败或过早结束任务。

本次修改目标：

1. 普通控制台输出、文件日志和推送通知按账号展示关键成功项、今日奖励和需要处理的问题。
2. 每个账号首次执行哪些社区任务只由该账号配置决定，不再使用初始米游币任务状态或奖励余额跳过首轮动作。
3. 社区签到、阅读、点赞和分享首轮执行后等待 3 秒并逐项复查；只有对应任务仍未完成时才重试，已经完成的任务立即停止。
4. 验证码平台返回临时异常时，等待 3 秒，重新创建米游社验证码并再次求解。
5. 统一业务任务的最大尝试次数，同时避免把网络请求重试与副作用动作重试混为一谈。
6. Cookie、Token、密码和含凭据 URL 等敏感信息在交互输入时使用 `*` 掩码，让用户能够确认输入和粘贴已经生效，同时不显示明文。

## 2. 交互、报告、日志与通知格式

### 2.1 展示原则

- 普通文本报告不再隐藏所有成功详情。
- 按账号分组，账号下按任务顺序展示关键结果。
- 国内游戏签到和 HoYoLAB 签到显示游戏、脱敏角色 UID、累计签到天数及今日奖励。
- 社区任务显示板块及米游币领取汇总。
- 阅读、点赞、取消点赞和分享的逐帖子记录不在普通报告中展开；普通报告按任务类型汇总，完整记录继续保留在 `run --verbose` 和 JSON 输出中。
- `Success` 和 `AlreadyCompleted` 均可进入“成功”区域，具体状态由文案区分。
- 跳过项仅在存在时增加“跳过”区域。
- 问题项按账号分组，并保留“认证失效”“需要验证码”“网络失败”“状态同步超时”“执行失败”等分类。
- 没有问题时显示“本次没有需要处理的问题。”，不输出空的失败区域。
- 通知仍遵守 `notifications.error_only`；启用仅错误通知时，不因新增成功详情而发送纯成功通知。

### 2.2 建议格式

```text
运行完成：成功 2，今日已完成 1，跳过 0，失败 0

成功：
- mys用户:10发满命胡桃
  / 国内游戏签到 / 原神 / ***1234：签到成功；今日奖励：原石 ×20；累计签到 7 天
  / 社区签到 / 原神：签到成功；已领取 50，还可领取 0，当前共 4219 米游币

- mys用户:XXXXXXXXXXX
  / 国内游戏签到 / 原神 / ***5678：今日已签到；今日奖励：摩拉 ×5000；累计签到 7 天
  / 社区签到 / 原神：今日已完成；已领取 50，还可领取 0，当前共 4219 米游币

本次没有需要处理的问题。
```

存在问题时：

```text
运行完成：成功 2，今日已完成 1，跳过 0，失败 1

成功：
- mys用户:10发满命胡桃
  / 国内游戏签到 / 原神 / ***1234：签到成功；今日奖励：原石 ×20；累计签到 7 天

本次需要处理的问题：
- mys用户:AAA
  / [状态同步超时] 社区签到 / 原神：已达到配置的最大尝试次数，服务端仍未确认签到任务完成
  处理方式：稍后重新运行；若持续出现，请检查米游社任务状态和账号网络。
```

### 2.3 游戏签到奖励

- 国内游戏签到与 HoYoLAB 签到在复查确认成功后获取奖励列表，并按确认后的 `total_sign_day` 选择今日奖励。
- 今日已经签到的角色也显示对应签到天数的奖励。
- 奖励获取失败不改变已确认的签到结果，显示“奖励详情获取失败”。
- 奖励列表不存在对应天数时显示“未找到第 N 天奖励”。
- 普通控制台输出、INFO 文件日志和通知使用相同的成功摘要。
- UID 继续使用脱敏值；不得记录 Cookie、SToken、验证码参数或通知凭据。

### 2.4 敏感信息输入反馈

#### 适用字段

以下交互式输入继续按敏感信息处理，但由“完全无回显”改为逐字符显示 `*`：

- 米游社和 HoYoLAB Cookie、SToken。
- 通知渠道的 Bot Token、应用 Token、SendKey、Secret 和 SMTP 密码。
- 云游戏 Token。
- 验证码平台端点、Webhook、代理 URL，以及其他可能在 URL 中携带用户名、密码或访问令牌的地址。

#### 交互规则

1. 每输入一个完整字符，在终端显示一个 `*`，不得显示原字符。
2. 粘贴一段内容时，按实际字符数连续显示对应数量的 `*`，让用户能够判断粘贴是否已经生效以及是否发生重复粘贴。
3. 掩码按 Unicode 字符计数，不能按 UTF-8 字节数输出多个 `*`。
4. 按一次退格键删除最后一个完整字符，并同步擦除一个 `*`。
5. 内容删除至空时，屏幕上不得残留掩码字符。
6. 按 Enter 后换行并立即显示“已接收，正在校验……”，避免格式校验或联网检查期间没有反馈。
7. `Ctrl+C`、`Ctrl+D` 等取消操作必须丢弃本次输入并恢复终端原始回显模式。
8. 已经保存在配置中的敏感值只显示 `<已配置>`，不得根据旧值长度预先显示星号。
9. 编辑时继续使用“留空保留、输入 `-` 清空”的规则；新输入的 `-` 在输入阶段同样只显示 `*`。
10. 非交互式标准输入保持静默，不输出提示或星号，避免污染 JSON 和脚本输出。

建议交互示例：

```text
请输入新的完整 Cookie（留空取消）> ********************************************************
已接收 Cookie，正在校验……
Cookie 格式有效；UID：***1234；SToken：已包含；MID：已包含

Telegram bot_token[未设置]> **********************************************
已接收 Bot Token，格式校验通过

代理 URL[<已配置>]（留空保留，- 清空）> ************************
已接收代理地址；协议：socks5h；包含身份认证
```

#### 安全边界

- 允许星号数量反映输入长度，但任何情况下不得提供切换明文显示的功能。
- 输入明文不得写入普通输出、日志、错误详情或 panic 信息。
- 格式验证消息只显示字段类型、校验结果和必要的脱敏信息。
- Cookie 可以显示脱敏 UID 以及是否包含 SToken、MID；不得显示 Cookie 键值。
- Token、密码和 Secret 不显示前缀、后缀或摘要。
- 代理和端点只显示协议及是否包含身份认证，不显示完整主机、路径、查询参数或用户信息。
- 校验失败、取消输入或完成持久化后，应尽快清空可变输入缓冲区。
- 若输入内容前后两半完全一致，或 Cookie 出现明显重复键，可提示“疑似重复粘贴”并要求重新输入；提示中不得包含原文。

## 3. 重试配置决策

### 3.1 不直接合并网络重试和业务尝试

`runtime.retry_count` 与 `runtime.game_checkin_max_attempts` 表面上都是次数，但语义不同：

- `retry_count` 控制连接失败、超时、HTTP 408/429/5xx 等网络层重试。
- `game_checkin_max_attempts` 控制可能产生副作用的签到业务总尝试次数，每次重提之前必须先确认服务端仍显示未完成。

若直接合并，同一个数值会同时放大网络请求和副作用动作次数；验证码流程还可能形成网络重试与业务重试相乘。因此本需求不建议直接合并两者。

### 3.2 建议调整

- 保留 `runtime.retry_count`，只表示网络层的额外重试次数；`0` 表示不进行额外网络重试。
- 将 `runtime.game_checkin_max_attempts` 泛化并重命名为 `runtime.task_max_attempts`，表示受保护业务流程的最大总尝试次数。
- `task_max_attempts` 默认值为 `3`，范围为 `1..=10`。
- 国内游戏签到、HoYoLAB 签到、社区签到和验证码重新生成求解均使用 `task_max_attempts`。
- 已映射的签到、阅读、点赞和分享任务均使用该值控制最大执行轮数；每轮提交后必须先复查对应任务，明确未完成才可进入下一轮。
- 旧配置中的 `game_checkin_max_attempts` 无损迁移为 `task_max_attempts`；缺失时使用默认值。
- 配置加载、`config setup`、示例配置、DaCapo/旧版配置迁移和使用说明同步更新。

建议配置：

```yaml
runtime:
  retry_count: 3
  task_max_attempts: 3
```

## 4. 社区签到流程

### 4.1 首轮执行条件

1. 每个账号是否执行社区签到，只读取该账号自己的 `tasks.bbs.enabled`、`tasks.bbs.sign` 和 `tasks.bbs.forums` 配置。
2. `sign: true` 时，每次程序运行都对该账号配置的社区板块执行第一轮签到；其他账号的开关和板块不得影响本账号。
3. 初始米游币任务查询如需保留，只能用于记录执行前基线、固定任务状态和米游币余额，不能用于跳过第一轮签到。
4. 初始响应缺少签到任务 `mission_id=58`、只包含 `62/64`、显示 `award_received=true` 或 `can_get_points == 0`，均不得单独作为跳过第一轮签到的依据。
5. 重复运行时仍允许执行第一轮；签到接口明确返回“今日已签到”等幂等结果时按已完成处理。

### 4.2 提交与事后复查

1. 记录签到前的 `already_received_points`、`can_get_points`、`total_points` 和固定任务状态；基线获取失败时不得伪造差值。
2. 对该账号配置的每个社区板块提交一次签到，第一轮总尝试次数记为 `1`。
3. 签到接口返回成功或幂等完成后等待 3 秒。
4. 重新查询米游币任务状态和米游币汇总，只判断本轮社区签到，不使用阅读、点赞、分享或 `62/64` 的状态替代签到结果。
5. 以下任一条件可确认社区签到完成：
   - 固定签到任务 `mission_id=58` 出现且 `award_received == true`；
   - 签到后 `already_received_points` 增加或 `can_get_points` 减少，能够确认本轮米游币到账；
   - 签到接口明确返回“今日已签到”等幂等完成结果。
6. 社区签到通常可能领取约 30–50 米游币，但不得把具体奖励数值硬编码为成功条件；报告应显示本轮实际差值和复查后的实时总数。
7. 复查成功但 `58` 仍未完成或缺失、米游币没有正向变化且接口未返回幂等完成时，视为本轮尚未确认；未达到 `task_max_attempts` 时重新执行签到。
8. 每次重新提交后继续等待 3 秒并复查，只在本轮结果仍未确认时进入下一轮。
9. 状态查询发生超时、连接错误、无效响应或认证失败时停止重提；查询失败不等于签到未完成。
10. 达到 `task_max_attempts` 后仍未确认，报告“状态同步超时”。

### 4.3 安全边界

- 每个账号、每个社区板块、每轮运行的提交次数不得超过 `task_max_attempts`。
- 第一轮由账号配置授权；第二轮及以后必须以前一轮动作完成后的成功复查结果为依据。
- 已确认完成的签到任务必须立即从待处理集合移除。
- 签到前后应与阅读、点赞、分享分开执行和复查，确保米游币差值可归因于当前任务。
- 重新提交后若再次触发验证码，进入第 6 节流程，但整体签到提交次数仍受 `task_max_attempts` 限制。

## 5. 社区任务首次执行与统一重试

### 5.1 账号配置决定首轮计划

每个账号独立从自己的 `tasks.bbs` 生成首轮任务计划：

- `sign: true`：执行该账号 `forums` 中的社区签到。
- `read: true`：执行阅读任务的正常首轮动作数。
- `like: true`：执行点赞任务的正常首轮动作数。
- `cancel_like: true`：仅在本账号点赞成功后执行取消点赞，不单独形成任务。
- `share: true`：执行一次分享任务。
- 配置为 `false` 的任务不得执行，也不进入复查或重试。

程序内部可以对本账号已启用任务采用确定性处理顺序，但不得假定所有账号都启用了签到、阅读、点赞和分享，也不得复用其他账号的执行计划。

### 5.2 固定任务 ID

米游币任务 ID 是固定的，不进行运行时动态推断或临时绑定：

| `mission_id` | 任务类型 | 自动执行与重试 |
|---:|---|---|
| `58` | 社区签到 | 按账号配置启用 |
| `59` | 阅读 | 按账号配置启用 |
| `60` | 点赞 | 按账号配置启用 |
| `61` | 分享 | 按账号配置启用 |
| `62` | 绑定游戏角色或修改个性签名，具体对应待确认 | 禁止自动执行 |
| `64` | 绑定游戏角色或修改个性签名，具体对应待确认 | 禁止自动执行 |

- `62` 和 `64` 分别属于“绑定游戏角色”和“修改个性签名”，但目前不确定两者的具体对应关系。
- `62/64` 不属于签到、阅读、点赞或分享，不能作为这四项任务的完成依据，也不能参与其重试判断。
- 即使以后确认 `62/64` 的具体对应关系，默认仍不得自动绑定角色或修改签名，避免改变用户账号资料。
- `62/64` 已完成时使用 `INFO` 或 `DEBUG` 记录；未完成时可使用 `WARN` 提示映射信息不完整，但不得触发动作。
- 其他未知固定 ID 必须保留原始编号和状态，不得猜测映射，也不得因为未映射任务存在就把已启用任务判定为完成。

### 5.3 统一执行模型

每个已启用任务独立维护首轮执行、复查状态和后续尝试次数：

1. 根据账号配置直接生成首轮计划，不使用初始 `award_received`、`happened_times`、任务缺失或米游币余额跳过首轮。
2. 逐项执行本账号已启用任务；一个任务执行后先等待 3 秒并完成复查，再进入下一个任务，避免米游币差值无法归因。
3. 首轮阅读和点赞执行正常目标数量；只有第二轮及以后才根据复查后的 `happened_times` 计算剩余动作。
4. 复查只检查刚执行的固定任务 ID 及该任务前后的米游币变化。
5. 对应任务完成或奖励到账后，立即标记成功并停止该任务的后续动作。
6. 复查请求成功且对应任务仍未完成或未确认，并且执行轮数未达到 `task_max_attempts` 时，执行下一轮。
7. 状态查询失败时停止该任务的后续动作；状态未知时不得盲目重试。
8. 达到 `task_max_attempts` 后仍未确认，报告该任务“状态同步超时”。
9. 多任务部分完成时，只继续处理尚未完成的任务；各任务轮数互不影响。

### 5.4 各任务重试规则

#### 社区签到

- 沿用第 4 节的配置驱动首轮、等待和事后复查流程。
- `58` 缺失不能阻止第一轮签到；第一轮后仍未确认时才允许重试。

#### 阅读

- 配置启用时先执行正常首轮阅读数量，不按初始 `happened_times` 缩减或跳过。
- 第二轮及以后根据复查后的 `happened_times` 计算剩余阅读次数。
- 优先选择本轮尚未读取的新帖子；可用帖子不足时记录原因，不无限重复同一帖子。

#### 点赞与取消点赞

- 配置启用时先执行正常首轮点赞数量，不按初始 `happened_times` 缩减或跳过。
- 第二轮及以后根据复查后的 `happened_times` 计算剩余点赞次数，优先选择尚未操作的新帖子。
- 启用 `cancel_like` 时，每次成功点赞后按本账号配置恢复为取消点赞状态；取消点赞不单独增加任务轮数。
- 验证码、认证失效或状态查询失败时停止后续点赞，避免未知状态下反复切换点赞状态。

#### 分享

- 配置启用时第一轮提交一次分享动作。
- 复查后仍未完成时允许下一轮再次分享，整体轮数不超过 `task_max_attempts`。

### 5.5 安全边界

- 自动执行和重试只适用于 `58`、`59`、`60`、`61` 四个已映射业务任务。
- 第一轮只由本账号配置授权；每次重试前必须有一次成功的事后状态查询。
- 初始任务状态仅是基线，不能用于跳过首轮，也不能把缺失任务解释为已经完成。
- `can_get_points == 0` 只能作为动作执行后的辅助完成依据，不能作为首次执行前的总开关。
- 已完成任务不得因其他任务未完成而再次执行后续轮次。
- 重试动作触发验证码时进入第 6 节流程，但任务总轮数仍不得突破 `task_max_attempts`。
- 日志应记录任务类型、当前轮数、最大轮数、复查前等待时间、实时米游币差值和剩余进度，不记录帖子或账号敏感参数。

## 6. 验证码平台处理

### 6.1 单次验证码尝试

每次验证码尝试都是完整且独立的一轮：

1. 调用 `createVerification` 创建新的米游社验证码，取得新的 `gt/challenge`。
2. 使用新参数调用验证码平台。
3. 验证码平台正常返回非空 `validate` 后，调用米游社验证码校验接口。
4. 校验成功后，携带服务端返回的 challenge 重试原动作。

### 6.2 可重新生成验证码的异常

以下任一情况视为本轮验证码求解失败：

- 验证码平台请求超时或连接失败。
- 验证码平台返回非成功 HTTP 状态。
- 响应 JSON 无效。
- 响应缺少非空 `validate`。
- `result` 为 `fail`、`failed` 或其他明确的非成功结果。

处理规则：

1. 当前验证码立即作废，不复用原 `gt/challenge`。
2. 若尚未达到 `task_max_attempts`，等待 3 秒。
3. 重新调用 `createVerification`，生成新的 `gt/challenge` 后再次调用验证码平台。
4. 验证码业务流程统一按 `task_max_attempts` 计数；验证码平台请求不得再叠加一套不可见的自动重试，避免实际请求数成倍增长。
5. 达到上限仍失败时，结果保持“需要验证码”，并注明实际尝试次数和最后一次失败原因。

### 6.3 不进入重新生成流程的错误

- 未配置验证码平台端点。
- 米游社 `createVerification` 返回明确的认证失效或不可恢复业务错误。
- 米游社验证码校验接口返回明确的认证失效或不可恢复业务错误。
- 验证码校验成功后，原动作仍要求验证码。

这些情况直接停止高风险动作，不继续生成验证码。

### 6.4 日志示例

```text
验证码平台返回异常，等待后重新生成验证码 attempt=1 max_attempts=3 delay_seconds=3 reason="result: fail"
验证码平台连续 3 次求解失败，已停止高风险动作 last_error="缺少非空 validate"
```

日志不得包含验证码端点中的访问凭据、完整 `gt`、`challenge` 或 `validate`。

## 7. 状态同步超时

“状态同步超时”只表示动作请求已经返回成功或已执行，但在配置的复查上限内仍未得到服务端任务状态确认，不等同于网络请求超时。

建议文案：

```text
[状态同步超时] 社区签到 / 原神：已提交 3 次并在每次提交后等待 3 秒复查，服务端仍未确认签到任务完成
[状态同步超时] 点赞 / 米游币任务：已执行 3 轮并在每轮后等待 3 秒复查，服务端仍未确认点赞任务完成
```

状态查询本身失败时应报告“网络失败”或具体接口错误，不得误报为“状态同步超时”。

## 8. 验收标准

### 8.1 报告

- 普通报告、INFO 文件日志和通知按账号显示关键成功项。
- 游戏签到成功和今日已签到均显示累计签到天数与今日奖励。
- 多账号、多游戏、多角色顺序稳定，UID 保持脱敏。
- 社区任务显示米游币已领取、还可领取和当前总数。
- 社区任务的最终币数使用动作后复查的实时数据，初始查询只能作为基线。
- 阅读、点赞、取消点赞和分享在普通报告中汇总，`run --verbose` 和 JSON 保留逐项记录。
- 有问题时按账号显示分类和处理建议；无问题时不显示空失败区域。
- 未实际执行且未获得完成证据的已启用任务不得显示“今日已完成”；找不到 `58–61` 不能视为全部完成。

### 8.2 社区签到

- 账号配置 `sign: true` 时，无论初始响应是否包含 `58`、是否只有 `62/64`，第一轮都调用签到接口。
- 两个账号使用不同 `sign` 和 `forums` 配置时，各自只执行自己的板块；关闭签到的账号不执行、不复查、不重试。
- 初始 `can_get_points=30`、`62/64` 已完成且 `58` 缺失：仍执行社区签到，不得直接报告“今日已完成”。
- 首次签到成功，3 秒后 `58` 完成或米游币从可领取 30 变为 0：只提交一次并报告实时领取结果。
- 首次复查请求成功但 `58` 仍未完成或缺失、米游币无变化：第二次提交后确认完成，最终成功。
- 签到接口返回“今日已签到”等幂等结果：按完成处理，不因重复运行误报失败。
- 达到 `task_max_attempts` 仍未确认：报告“状态同步超时”。
- 任一次事后状态查询失败：停止重提，签到接口调用次数不再增加。
- 签到完成后不受阅读、点赞、分享或 `62/64` 状态影响，也不再进入后续签到轮次。

### 8.3 其他社区任务与固定 ID

- 每个账号只对自身配置为 `true` 的阅读、点赞和分享执行第一轮；关闭的任务不执行。
- 初始任务状态显示完成、缺失或存在进度时，均不能跳过已启用任务的第一轮。
- 阅读和点赞第一轮执行正常目标数量；首次复查仍未完成时，下一轮只补足复查后的剩余次数。
- 对应任务确认完成后，该任务的动作调用次数不再增加。
- 每个任务的执行轮数不超过 `task_max_attempts`。
- 启用 `cancel_like` 时，每次点赞成功后均按本账号配置恢复取消点赞状态。
- 多任务部分完成时，仅重试尚未完成的任务，各任务轮数独立。
- 状态查询失败时不重试动作；达到执行轮数上限仍未确认时报告“状态同步超时”。
- `62/64` 分别属于绑定游戏角色和修改个性签名，但具体对应关系未知；两者不得自动执行，也不得影响 `58–61` 的完成判断。
- `62/64` 已完成时不输出 WARN；其他未知固定 ID 会被记录，但不会触发动作或重试。

### 8.4 验证码

- 首次验证码平台异常、第二次使用全新 `gt/challenge` 成功：最终继续原动作。
- 每次异常后等待 3 秒，最后一次失败后不再等待。
- 连续异常时，`createVerification` 和验证码平台调用次数均等于 `task_max_attempts`。
- 网络超时、无效 JSON、缺少 `validate`、`result: fail` 均覆盖测试。
- 不存在网络重试与验证码业务重试相乘的情况。
- 日志包含尝试次数、等待秒数和脱敏后的原因，不包含敏感参数。

### 8.5 配置兼容

- 旧 `game_checkin_max_attempts` 可迁移为 `task_max_attempts`。
- 新旧配置缺少业务尝试字段时使用默认值 `3`。
- `task_max_attempts` 的边界值和非法值有配置校验测试。
- 文档、示例配置、交互设置和适配入口使用一致的字段名与语义。

### 8.6 敏感信息输入

- 单字符键盘输入显示一个 `*`，不显示原字符。
- 整段粘贴立即显示与字符数一致的星号，不丢失输入。
- 多字节 Unicode 字符只对应一个 `*`，退格一次可完整删除。
- 连续退格至空会清除全部星号，内部输入缓冲区也为空。
- 按 Enter 后立即显示已接收和校验状态，不必等待联网操作完成。
- 已配置的敏感值只显示 `<已配置>`，不泄露旧值长度。
- 非交互式输入不输出提示或星号。
- 取消、EOF、读取错误和异常退出路径均恢复终端回显。
- Cookie、Token、密码、代理凭据和验证码端点均不会出现在 stdout、stderr、日志或错误详情中。
- 疑似重复粘贴只输出脱敏警告，并允许用户安全地重新输入。

## 9. 非目标

- 不修改通知渠道协议或各推送服务的鉴权方式。
- 不在普通报告中展开每个帖子 ID、标题或每次底层 HTTP 请求。
- 不对状态未知的副作用动作进行盲目重试。
- 不因为奖励详情获取失败而把已经确认的游戏签到改判为失败。
- 不提供敏感输入明文显示或临时切换明文的功能。
